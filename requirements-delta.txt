# DNABERT-2 LAMBDA replication — Delta-AI (NCSA) install, GH200 / aarch64.
#
# WHY THIS FILE EXISTS: the main requirements.txt targets Biowulf x86 (Python 3.8,
# torch==1.13.1). Neither installs on a GH200: it is aarch64 + Hopper (sm_90), and
#   - PyTorch dropped Python 3.8 after torch 2.4.1, and
#   - aarch64 CUDA wheels only exist for Python >= 3.9 (3.10+ in practice), and
#   - torch 1.13.1 has no aarch64 CUDA wheel at all and no sm_90 support.
# So Delta-AI needs Python 3.11 and a modern CUDA torch.
#
# BUILD (confirmed working 2026-06-24 on a GH200 node; driver 590.48.01 / CUDA 13.1):
#   conda create -y -p /work/hdd/bfzj/llindsey1/conda/envs/dnabert2_env python=3.11
#   conda activate /work/hdd/bfzj/llindsey1/conda/envs/dnabert2_env
#   pip install --no-cache-dir torch          # -> torch 2.12.1+cu130 (cuda 13.0, is_available True)
#   pip install --no-cache-dir -r requirements-delta.txt
#   pip uninstall -y triton                   # REQUIRED — see "Triton / flash-attention" below
#
# Triton / flash-attention (REQUIRED on GH200):
#   DNABERT-2's remote model code ships a Triton flash-attention kernel
#   (flash_attn_triton.py) that calls tl.dot(..., trans_b=True). `trans_b` was
#   removed from tl.dot in Triton 3.x (bundled with torch 2.12), so the kernel
#   fails to compile at the first training step:
#       TypeError: dot() got an unexpected keyword argument 'trans_b'
#   bert_layers.py already wraps the Triton import in try/except and falls back to
#   an exact pure-PyTorch attention path when the import fails. So the fix is to
#   REMOVE Triton (`pip uninstall -y triton`); the model then uses PyTorch
#   attention (same math; Flash is only a speed optimization). Triton is not
#   needed for eager-mode training/inference (only torch.compile, unused here).
#   Downgrading Triton instead is NOT viable: Triton 2.x predates Hopper/sm_90
#   and is ABI-incompatible with torch 2.12.
#
# torch is installed SEPARATELY (line above), not pinned here, because the correct
# wheel is architecture/CUDA-specific (plain `pip install torch` resolves to the
# aarch64 CUDA build on Python 3.11). Verify before training:
#   python -c "import torch; print(torch.__version__, torch.version.cuda, torch.cuda.is_available())"
#
# transformers/peft/accelerate are kept at the repo's original pins (DNABERT-2's
# remote model code expects this API); numpy/scikit-learn/einops/omegaconf/evaluate
# are left unpinned so pip picks py3.11/aarch64 wheels instead of building old ones.
